# Mask Source Strategy

The repository should not be framed as replacing Brieflow or CellPose with SSL. Instead, it should accept multiple mask sources and evaluate SSL features downstream of whichever mask source passes QC.

## Supported mask-source categories

Use the following controlled labels in configs and metadata:

| Label | Meaning | Typical use |
|---|---|---|
| `brieflow` | Masks generated by Brieflow workflow | Existing lab pipeline |
| `cellpose` | Masks generated by CellPose or CellPose-SAM | General cell/nuclei segmentation |
| `sam` | Segment Anything style prompts/adaptation | Exploratory/manual-assisted segmentation |
| `manual` | Human-curated ground truth | Evaluation, training, validation |
| `stardist` | StarDist-style nuclei/cell segmentation | Dense nuclei/cell objects |
| `other` | Any external mask source | Prototypes and imports |

## Required metadata

Every phenotype output should record:

- `meta_mask_source`
- `meta_mask_source_detail`
- `meta_segmentation_model`
- `meta_segmentation_qc_status`
- `meta_ssl_role`

These fields make it clear that SSL features were extracted **inside existing masks** rather than used as a complete segmentation replacement.

## QC logic

Preferred order:

1. Validate image/channel/Z-stack quality.
2. Validate mask source and segmentation QC.
3. Validate ProCode on/off decoding and channel unmixing.
4. Extract classical and/or SSL features.
5. Benchmark phenotype separability, control behavior, replicate consistency, and batch leakage.

## Large ground truth

For several-gigabyte ground-truth masks/images, keep only manifests in GitHub. Store raw images, masks, and annotations externally with DVC, Git LFS, lab storage, or a public archive. Split train/test by image, well, plate, or replicate rather than by individual cell rows.
